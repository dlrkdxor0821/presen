# ABA · Libi — 향후 개발 계획 (Roadmap)

> **리비(Libi)** — 주행로봇 + 로봇팔이 결합된 모바일 매니퓰레이터. 중앙관제(**FMS**)가 여러 UI의 명령을 받아
> **배차(누가) + 교통협상(주행 충돌 회피)** 으로 멀티 로봇을 지휘하는 도서관 협동로봇.
>
> 본 문서는 `arte_docs`(시스템 요구사항·시나리오·아키텍처·ERD·인터페이스 명세)와 Jira 진행상황을 근거로,
> **Sprint 4~8 (2026.07.06 – 08.07)** 및 **데모데이(08.09~08.10)** 까지의 상세 개발 계획을 정의한다.

## 1. 현재 상태 스냅샷 (Sprint 4 기준)

인터페이스 명세의 구현 마커로 본 착수 지점. **✅ 구현 / 🟡 스텁 / 📐 설계(코드 미구현)**

**이미 동작(✅)**
- `aba_service`(:8000) `/tasks` → ROS2 `/fms/submit_task` 브리지
- `console`(:8001) 관제 대시보드 — 상태 400ms 폴링, 배차/교통 플러그인 런타임 전환, navgraph 편집
- **FMS `fleet_node`** — 배차 + 교통협상 + 상태머신, `PathRequest` 기반 슬롯카 **실주행**, 도착 판정(`kArrive=0.35m`), `/fms/task_states` 진행 보고
- 배차·교통 **pluginlib** — GreedyCost/Hungarian/Auction/… × EdgeNodeLock/Priority/ORCA/… 런타임 교체

**부분/스텁(🟡)**
- `Navigate` / `PerformAction` 액션 — 정의·스텁만 존재, **fleet이 아직 호출하지 않음**
- `drive_node`(주행 컨트롤러) — 단독 스켈레톤

**미구현(📐, 이번 로드맵의 핵심 개발 대상)**
- **Handy(로봇팔) 컨트롤러** — 코드 없음(README 스텁). 파지/배치/정렬 전부 신규 구현 필요 → **최대 리스크**
- `ai_service` — 비전(책/사람/마커)·추천·대화·STT. UDP 이미지 파이프라인 미구현
- **실시간 채널** — WebSocket / MJPEG / UDP 없음, 현재 전부 HTTP 폴링
- 웹서비스 / 회원·사서 브라우저 UI, DB(MariaDB `labi`), 인증 — 설계만

**Jira Sprint 4(active) 진행 중 항목** — 이 위에서 다음 스프린트로 이어진다.
- 정밀주행: ArUco Marker(AR-52), Line tracing(AR-53) *In Progress*
- 딥러닝 모델: 바구니(AR-31), 핑키+사람 모형(AR-58) *In Progress*
- FMS sim: slotcar plugin(AR-59) *In Progress*
- Server 구축(AR-45) *To Do*
- 로봇팔 파지 후 배치: 기술조사(AR-60) *In Progress* / 실물 테스트(AR-61) *To Do*
- 추종(tracking): 기술조사(AR-62) / 실물 테스트(AR-63) *To Do*
- 분류 기능 기술조사(AR-26), 바구니 grab 기술조사(AR-49)

## 2. Sprint 4 (W1, 7.6 – 7.12) — v0.2 · 전체 20% *(진행 중)*

> **목표: 서버·FMS·정밀주행·딥러닝 모델을 병렬로 구현 착수하고, 주행 중심의 "이동 시나리오"를 처음으로 통합한다.**

현재 진행 중인 스프린트. 파지·시나리오 확장(Sprint 5~)의 토대를 만드는 단계로, 각 축의 **뼈대 구현 + 다음 단계 기술조사**가 병렬로 진행된다.

### 개발 항목

**(1) Server 구축 — 📐→🟡**
- `aba_service`(:8000)·`console`(:8001) 위에 태스크 접수·상태 연동 서버 골격 정비
- `/tasks` → `/fms/submit_task` 브리지 안정화, 상태 폴링 경로 확인
- *근거: Jira AR-45 (To Do → 착수)*

**(2) FMS 구현 (sim) — ✅ 강화**
- Gazebo sim 상 slotcar plugin으로 **실주행** 재현 (배차 + 교통협상 + PathRequest)
- navgraph 편집·재적재 경로 검증
- *근거: Jira AR-59 (In Progress)*

**(3) 정밀주행 착수 — 🟡**
- ArUco Marker(AR-52)·Line tracing(AR-53) 1차 구현, Nav2 params·PID 제어 튜닝 시작
- *Sprint 5에서 마감 → 파지의 선행 조건*

**(4) 딥러닝 모델 구축 — 🟡**
- 바구니 인식 모델(AR-31), 핑키 + 사람 모형 인식 모델(AR-58) 1차 학습
- 온디바이스 vs AI 서버 추론 성능 비교(기존 조사 AR-14/AR-41) 반영
- *진열·추종·순찰 인식의 기반*

**(5) 다음 단계 기술조사**
- 로봇팔 파지 후 배치 가능성 검토·관련 기술(AR-60) + 실물 테스트(AR-61)
- 추종(tracking) 기술조사(AR-62) + 실물 테스트(AR-63)
- 분류 기능(AR-26)·바구니 grab(AR-49) 조사
- *→ Sprint 5 로봇팔/추종 구현 착수의 전제*

**(6) 이동 시나리오 1차 통합**
- `/tasks` → FMS 배차 → 교통협상 → `PathRequest` → 주행 → **도착 통지** 관통
- 아직 팔·인식 없이 **주행 레벨** 만의 end-to-end 흐름 확보

### 완료 조건 (DoD)
- [ ] 이동 시나리오: `접수 → 배차 → 주행 → 도착 통지` 무중단 관통
- [ ] 정밀주행: ArUco/Line tracing 기본 동작 확인 (마감은 S5)
- [ ] 딥러닝: 바구니·사람 인식 모델 1차 학습 결과 확보
- [ ] 기술조사: 로봇팔 파지·추종 실물 테스트로 S5 구현 착수 근거 확보

### 산출물
- **Integration Test Report (v0.2)** — 서버·FMS·정밀주행·모델 구현 착수 및 이동 통합 결과

### 선행 의존성 · 리스크
- ⚠️ 이번 스프린트의 **파지·추종 기술조사(AR-60~63)** 가 Sprint 5 구현 착수의 전제 → 조사·실물 테스트 지연 시 S5 로봇팔 일정에 직접 파급.
- 정밀주행(ArUco/Line)은 S5로 이어져 **마감** → 여기서 기본기가 잡혀야 파지 도킹이 성립.

## 3. Sprint 5 (W2, 7.13 – 7.17) — v0.3 · 전체 50%

> **목표: 로봇팔을 처음으로 실제 동작시켜 "픽업 → 이송 → 진열"을 단일 로봇 end-to-end로 완성하고, 정밀주행을 마감한다.**

이번 스프린트가 전체 로드맵의 분기점이다. 지금까지의 주행(FMS)에 **매니퓰레이션(팔)** 을 붙여 리비를 "모바일 매니퓰레이터"로 만든다.

### 개발 항목

**(1) 정밀주행 마감 — 🟡→✅**
- ArUco Marker 기반 도킹 정렬 완성 (서가/데스크 앞 정위치 정지)
- Line tracing 안정화 (좁은 통로·서가열 진입)
- Nav2 params + PID 튜닝 종료 → 이후 모든 시나리오의 **정지 정확도 기준** 확정
- *근거: 파지 성공률은 도킹 정확도에 종속 → 픽업보다 선행 완료 필수*

**(2) 로봇팔(Handy) 컨트롤러 신규 구현 — 📐→🟡/✅**
- `libi_handy_controller` 실코드 착수 (현재 코드 전무)
- Drive가 팔을 지휘하는 계약 구현: `PerformAction(action=pick|place|align, params)`
- 그리퍼 카메라 인식 → 접근 → 파지 → 들어올림 시퀀스
- **재시도 정책**: 픽업 실패 시 최대 3회 재시도, 실패 시 종료·UI 통지·작업내역 저장 (시나리오 규정)

**(3) 이송 시나리오 (delivery) — 서가 → 특정 장소**
- 흐름: `/tasks(delivery)` → FMS 배차 → 주행(서가 도킹) → **픽업** → 주행(목적지) → **배치** → 완료 통지
- DB에서 책 출발지(서가 위치) 조회 → Nav2 목표(`rc_map_locations`)로 변환

**(4) 진열 시나리오 (place) — 바구니 책 → 해당 서가**
- 현재 로봇 바구니의 책 인식 → 해당 청구기호 서가로 이동 → 배치
- *바구니 인식은 딥러닝 모델(AR-31, AR-58) 결과 연동*

**(5) 추종(tracking) 기본 구현**
- `drive → ai_service(이미지) → 연산결과 → drive 제어` 파이프라인 1차
- 대상 인식·거리 유지 기본 동작 (완성/실패복구는 Sprint 6에서 마감)

### 완료 조건 (DoD)
- [ ] 정밀주행: 서가·데스크 앞 도킹 오차 목표치 이내, ArUco/Line 반복 성공
- [ ] 로봇팔: 정해진 위치의 책 1권 파지 → 지정 위치 배치 실물 성공
- [ ] 이송 시나리오: `접수 → 이동 → 픽업 → 이동 → 배치 → 완료 통지` 무중단 관통 1회 이상
- [ ] 진열 시나리오: 바구니 책 인식 → 서가 배치 관통
- [ ] 추종: 대상 추종 주행 기본 동작 확인

### 산출물
- **Integration Test Report (v0.3)** — 이송/진열/파지 통합 테스트 결과

### 선행 의존성 · 리스크
- ⚠️ **로봇팔 코드 전무 → 이번 로드맵 최대 리스크.** 파지 실물 테스트(Jira AR-61)를 스프린트 초반에 집중 배치, 실패 시 파지 자세/그리퍼 조건을 좁혀 **시연 가능한 최소 케이스**부터 확보.
- 의존 순서: **정밀주행(도킹) → 픽업 → 이송/진열**. 도킹 미완 시 파지 성공률 급락하므로 (1)을 (2)보다 우선.

## 4. Sprint 6 (W3, 7.20 – 7.24) — v0.4 · 전체 80%

> **목표: 분류(arm_sort)와 사람 상호작용(짐꾼·길잡이)을 붙이고, GUI ↔ 서버를 폴링에서 실시간 채널로 전환한다.**

### 개발 항목

**(1) 분류 시나리오 (sorting / arm_sort) — 수거함 → 분류장소 → 분류행위**
- 수거함의 책 인식 → 파지 → 분류 기준(청구기호/카테고리) 판정 → 분류 위치로 이동·배치
- Sprint 5의 파지·배치 능력 위에 **다중 대상 반복 처리** 로직 추가
- *근거: Jira 분류 기술조사(AR-26), 딥러닝 분류 모델*

**(2) 짐꾼(추종) 완성 — follow**
- Sprint 5 추종 기본 위에 **실패 복구 로직** 구현:
  `실패 시 10초 대기 → 좌우 30° 탐색 → 180° 회전 후 30° 탐색 → 재실패 시 순찰모드 전환`
- 시작/정지 제어(`/robots/{id}/follow`), 사용자 동행 상태 표시

**(3) 길잡이 (guide)**
- 목적지 DB 조회 → 로봇 앞장 → (추종 로직으로) 사용자 동행 검증 → 도착 시 UI 통지
- GUI guidePhase: `idle → guiding → (requesterLost) → completed/failed/cancelled`

**(4) GUI ↔ 서버 실시간 연동 — 📐→✅ (폴링 탈피)**
- `/ws/robot-state`, `/ws/task-state` WebSocket 방송 (로봇 위치·상태·태스크 전이)
- `/ws/video/{robot}` MJPEG (그리퍼/전방 카메라)
- `UDP(Image→ai_service)` perception 프레임 파이프라인 정식화
- 회원/사서/로봇 QT UI가 **실시간**으로 로봇 위치·작업 상태를 반영

### 완료 조건 (DoD)
- [ ] 분류: 수거함 다권 → 기준별 분류 배치 관통, 오분류 처리 경로 확인
- [ ] 짐꾼: 대상 상실 → 탐색 → 복구/순찰전환 시퀀스 실물 동작
- [ ] 길잡이: 목적지 안내 → 동행 검증 → 도착 통지 관통
- [ ] 실시간: 폴링 없이 WebSocket으로 로봇 상태/태스크 갱신, 영상 스트림 표시

### 산출물
- **Integration Test Report (v0.4)** — 분류·짐꾼·길잡이·실시간 연동 결과

### 선행 의존성 · 리스크
- 분류·짐꾼·길잡이 모두 **인식(ai_service)** 에 의존 → 비전 파이프라인(UDP) 안정성이 관건.
- ⚠️ 추종 실물 안정성(조명·가림·사람 밀집)이 낮을 수 있음 → 실패 복구 로직으로 방어, 데모 환경 통제.
- 실시간 채널 전환은 **CPU 부하 증가** 유발 → Sprint 8 부하 대응 항목과 연결해 미리 계측.

## 5. Sprint 7 (W4, 7.27 – 7.31) — v0.5 · 전체 100%

> **목표: 순찰·야간 보안을 구현해 시나리오 8종을 모두 채우고, 전체를 하나로 통합한 뒤 멀티로봇으로 검증한다.**

### 개발 항목

**(1) 순찰 (patrol)**
- 정해진 경로 주행 7분 30초 → 자동 복귀 (SR-16/19)
- 순찰 중 **사람 감지 → 추적 · 녹화 → 관리자 통지(영상 포함)**
- 추종/추적 로직(Sprint 5~6) 재사용

**(2) 야간 보안 (security)**
- 운영모드 전환(`day | night`), 침입 알림 목록, 침입 현장 녹화 영상 조회
- 사서 UI 보안 화면 연동

**(3) 전체 시나리오 통합**
- 이송·픽업·진열·분류·짐꾼·길잡이·순찰·보안 8종을 **단일 시스템에서 순차/동시 실행**
- 상태머신 전이 검증: 배터리 부족 시 `returning → 충전 도킹, 이동 중 명령 무시` 등 예외 흐름 포함

**(4) 멀티로봇 검증**
- FMS **배차 + 교통협상**을 복수 로봇으로 실검증 (`fleet ↔ Drive only` 원칙 유지)
- 배차 전략(Hungarian/Auction 등) × 교통 전략(EdgeNodeLock/Priority/ORCA 등) 조합 비교
- 교차·병목 지점에서 충돌 회피(대기/우회) 동작 확인

### 완료 조건 (DoD)

## 10. 리스크 & 의존성 관리

**핵심 의존성 체인**
```
정밀주행(도킹 정확도)  →  픽업·파지  →  이송/진열  →  분류
ai_service(비전)  →  추종  →  짐꾼 / 길잡이 / 순찰-추적
FMS 배차·교통(✅ sim)  →  멀티로봇 실검증
실시간 채널(WS/UDP)  →  GUI 실시간  ↘
비전 추론 동시 구동     →              CPU 부하 대응(S8)
```

**Top 리스크**
1. **로봇팔(Handy) 코드 전무** — 파지 실물 난이도 최상. → S5 초반 실물 테스트 집중, 최소 파지 케이스부터 확보.
2. **추종 실물 안정성** — 조명·가림·혼잡에 취약. → 실패 복구 로직(10초→탐색→순찰전환)으로 방어, 데모 환경 통제.
3. **멀티로봇 실물 검증** — 현재 sim/slotcar 중심. → S7에 sim 완전검증 + 실물 축소 데모 병행.
4. **CPU 부하** — 실시간 채널 + 비전 추론 동시 구동. → 온디바이스 vs 서버 추론 배치 결정(S8), 해상도·주기 튜닝.
5. **일정 압축** — 기능 100%가 S7, 안정화가 S8뿐. → S5~S7 매 스프린트 Integration Test Report로 조기 결함 발견, 기능 동결선(S8) 엄수.
