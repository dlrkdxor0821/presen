## p2 · Topic — 사서를 대체하지 않는 협동로봇 (약 40초)

**주행로봇과 로봇팔을 결합한 모바일 매니퓰레이터**로, 서가에서 책을 직접 집어 배달·반납·정렬합니다.

이번 주 보고는 이 기능을 구성하는 다섯 축 — **주행, 로봇팔, 딥러닝, 관제, GUI** 순서로 말씀드리겠습니다.

## p3 · 주행 Navigation (약 45초)  `[주행]`

먼저 **주행**입니다.
현재 저희 도서관 맵 환경에서 **Nav2 패키지로 자율주행에 성공**했습니다. 컨트롤러 서버는 **RPP**를 사용했고, 전반적인 파라미터를 튜닝했습니다.
다음은 **ArUco 마커 기반 정밀 제어**인데요, 충전소 복귀 도킹을 ArUco로 정밀하게 맞추는 걸 목표로, **3D 포즈(X·Y·Yaw)로 전환해 정면 수직 도킹**을 시도하고 있습니다. 다만 아직 완전히 원활하진 않은 상태입니다.
향후에는, 우선 기능 구현을 위해 **Line tracing을 먼저 적용**하고 이후 ArUco로 대체할 계획입니다. 인원은 지금 로봇팔에 대부분 투입돼 주행을 한 명이 담당하고 있는데, **로봇팔 기술조사를 빠르게 마무리하고 주행 쪽으로 재배치**할 예정입니다.

## p4 · 로봇팔 Robot Arm (약 1분 20초) `[로봇팔]`

이번 주 가장 공을 많이 들인 **로봇팔**입니다. **기술조사 → 현재 채택 기술 → 향후 계획** 순으로 말씀드리겠습니다.

**기술조사**부터. Perception은 — YOLO는 책 구분이 어려워 제외, **OCR은 신간 대응까지 되어 채택**, QR은 인식률이 너무 낮아 제외, **ArUco는 채택**했습니다. Depth 카메라는 최소거리 15cm 제약이 있어 미사용하고 더 나은 방법을 택했습니다. 모션 제어는 move L·move J·jog 방식을 검토했습니다.

**현재**는 **ArUco 마커 방식**으로 제어합니다. rvec·tvec 기반으로 위치 좌표를 계산하고, 카메라–그리퍼 오프셋은 **Hand-eye Calibration**으로 잡았습니다. 핵심은 **‘One-shot pose grasp’** — 계속 갱신·보정하는 대신 초기에 좌표를 한 번 찍고 이동하는 방식인데요, 이 방식으로 **도서 파지에 성공**했습니다. 지금은 성능 개선을 위해 파라미터를 조정 중입니다.

**향후**에는 **압전센서**를 붙여 파지 성공 여부를 판단하고, 반복 실험으로 카메라 오프셋·최종 접근 거리·후퇴 거리를 잡을 예정입니다. 그리고 **place 동작**인데, pick은 PBVS를 유지하되 place는 **IBVS 적용을 검토**하고 있습니다. 현재 PBVS는 이미지 갱신→제어→정지를 반복하는 동기 방식이라 팔 움직임이 끊기는 문제가 있어서, **비동기 제어 루프로 영상 처리와 제어를 병렬 수행**하는 방식을 보고 있고요. 최종적으로는 **ArUco 없이 비전만으로 책꽂이 빈칸을 실시간 추정해 place까지 수행**하는 걸 목표로 합니다.

## p5 · 딥러닝 모델 (약 45초) `[딥러닝]`

**딥러닝 모델**입니다.
구현된 모델부터 보면 — **YOLOv8n**을 635장으로 학습해 ep72가 베스트였고, Teachable Machine에서 전환했습니다. 성능은 엣지 Pi4에서 **4.96FPS·CPU 85%·67.6℃**인 반면 서버에서는 **19FPS·21ms**라, **서버로 오프로드**하기로 했습니다.
이 외에 **EasyOCR**로 한국어 책 분류를 Pi에서 실시간 처리하고, **Labi Bot**은 Ollama qwen3 1.7B 경량 RAG로 **4개국어**를 지원합니다.
아직 미구현인 건 **바구니 인식 모델, 핑키+사람 모형 추종 인식, 도서 분류 커스텀 YOLO, 분류 모델 조사**입니다.

## p6 · 관제 Fleet Control (약 1분) `[관제]`

**관제, Fleet Control**입니다. 왼쪽이 전체 시스템 아키텍처 흐름이고, 핵심 알고리즘 두 가지를 소개드리겠습니다.

**배차**는 **Auction/SSI** 방식입니다. 유휴 로봇들이 목적지까지 **Dijkstra 경로비용으로 입찰**하고 **최저가가 낙찰**됩니다. 가용 로봇이 없으면 거절(503)하고요. pluginlib 구조라 전략 교체가 가능합니다.

**교통 협상**은 **노드+엣지 예약** 방식으로, 반대방향 엣지를 잠가 **정면충돌을 차단**합니다. 이동 요청마다 **GRANT·WAIT·DEADLOCK** 세 결정으로 흐름을 제어하고, 대기가 사이클을 이루면 **wait-for 그래프를 DFS로 훑어 교착을 감지**해 **우선순위가 가장 낮은 로봇이 양보**합니다.

전체 흐름은 아래처럼 **태스크 접수 → 배차 → 교통 협상 → 경로 실행 → 팔 동작** 순으로 이어집니다.

## p10 · 애로사항 3건 (약 55초)

이번 주 **애로사항은 세 가지**입니다.

**첫째, Pi4의 CPU 점유율**입니다. 순수 bringup만 60~80%인데 ArUco·UDP 영상 송출이 붙으면 **97%를 넘고**, 여기에 nav2 주행 토픽까지 받으면 더 악화될 우려가 있습니다. 대응으로 **Behavior Tree로 프로세스 사용 구간을 나누고, perception을 AI 서버로 오프로딩**하고, 노드를 경량화하려 합니다.

**둘째, 길잡이(GUIDE) 기능**입니다. 현재 센서 — 2D 라이다와 파지용 D435로는 **전방 사용자·경로 인식 화각이 부족**합니다. **전방 인식용 카메라를 추가**하고 추종 모델과 연동할 계획입니다.

**셋째, 바구니 모델 선정**이 아직 미확정이라 **분류·이송 시나리오가 블로킹**되고 있습니다. 인식 모델 후보를 비교·선정하고 학습데이터를 보강해 확정하겠습니다.
